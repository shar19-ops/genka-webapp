# 原価計算表アプリ（genka-webapp）

雄電社の紙/Excelベース「原価計算表」をデジタル化するWebアプリ。既存の帳票PDF（縦書き・特殊レイアウト）を取り込んでブラウザ上で編集し、今後の支出予定や追加工事予算を入力すると自動で小計・合計・粗利率を再計算、A3用紙相当の印刷/PDF出力を行う。社内の担当者・確認者・承認者による検印フローにも対応。

- **公開URL（GitHub Pages）**: https://shar19-ops.github.io/genka-webapp/
- **ローカル起動**: `npm install` → `npm start` → http://localhost:5173

## 特徴・技術構成

- Node.js/Express（ビルドステップなし） + バニラJS（フレームワーク未使用）
- サーバーなし（GitHub Pages）でも、ローカルサーバーありでも同じコードで動く二重構成
- PDF.js（`public/vendor/pdfjs/`に物理コピー済み）でPDFをブラウザ内で直接解析。外部送信なし
- データはブラウザのlocalStorage、または任意の場所へのファイル保存（`.json`）で管理。サーバー側データベースは持たない

## ファイル構成

| ファイル | 役割 |
|---|---|
| `server.js` | Express静的配信 + `/api/fetch-pdf?url=`（CORS回避プロキシ、ローカル実行時のみ有効） |
| `public/index.html` | 画面骨格（取込パネル／メイン表示／印刷用シート／検印欄） |
| `public/pdf-parser.js` | `GenkaPdfParser.parsePdf()` — PDF.jsで座標ベース解析 |
| `public/app.js` | メインロジック（描画・計算・保存・印刷・検印フロー・ファイル入出力） |
| `public/date-stamp.js` | 丸型デート印描画ライブラリ（`DateStamp.draw(ctx, cx, cy, options)`） |
| `public/style.css` | 画面表示＋印刷（`@media print`）両対応スタイル |
| `public/manifest.json` / `public/sw.js` | PWA化（ファイルのダブルクリックで自動起動するための仕組み） |
| `public/vendor/pdfjs/` | pdf.js本体・cmapsの物理コピー（`node_modules`から配信するとGitHub Pagesで動かないため） |
| `.github/workflows/pages.yml` | pushのたびに`public/`をGitHub Pagesへ自動デプロイ |

## 主要機能

### 1. PDF取込・自動計算
- 座標(x, y)ベースでPDFのテキストを行・列に再構成して解析（`pdf-parser.js`）。帳票のバージョン差異（数値列が4〜6列）に対応
- 差引後残額 = (予算残額 + 追加工事予算) − 今後の支出予定、実行/追加工事/最終の各粗利率などを自動計算
- 取込プレビュー画面でメタ情報（工番・得意先名・件名等）を確認・修正してから確定できる
- 同じ工番への再取込時は、工番不一致なら中断する安全チェックあり

### 2. A3印刷/PDF出力
- `@page { size: A3 landscape; margin: 12mm; }`。ネイティブ`<thead>`でページをまたいでもヘッダーが自動繰り返しされる構成

### 3. 検印フロー（担当者→確認→承認）
- 押印欄は6枠固定（担当者1・確認4・承認1）。ワークフロー的なソフトロック（本人確認の強制力はない、詳細は下記「既知の制限」参照）
- 確認者が何人挟むかは可変（「もう1人確認を挟む」or「承認へ進む」を選択、余った確認欄は自動スキップ）
- 差し戻し機能あり（1つ前の実段階に戻る、往復とも押印データはクリアされる）

### 4. ファイルによる受け渡し（Teams/メール送付ワークフロー）
サーバー・共有ストレージを使わず、以下の流れで複数人の検印フローを成立させる:
1. 担当者が入力 →「確認者へ提出」→ ファイル（`工番_原価計算書_日付_苗字.json`）が保存される → Teams/メールに添付して次の人へ送付 → アプリが閉じる（提出者本人のローカルは提出前の状態のまま維持）
2. 受け取った人がファイルを開く（アプリの「ファイルを開く」ボタン、または対応ブラウザでPWAインストール済みならファイルのダブルクリックで自動起動）→ 続きを入力 → 同様に次へ提出

## デプロイの仕組み（GitHub Pages）

- `main`ブランチにpushすると`.github/workflows/pages.yml`が`public/`フォルダをアーティファクトとしてアップロードし自動デプロイする（`actions/upload-pages-artifact` + `actions/deploy-pages`）
- GitHub Pagesは`public`のような任意サブフォルダを直接指定できないため、ワークフロー経由のデプロイ方式（`build_type=workflow`）を採用している
- サーバー機能（`/api/fetch-pdf`、URLからのPDF取込プロキシ）はGitHub Pagesでは動作しない。起動時に自動検知し、利用できない場合は「URLから読み込む」タブを自動的に非表示にする（ローカルで`npm start`した場合は引き続き使える）

## 開発上の重要な注意点

- **A3横（landscape）は幅420mm×高さ297mm**。高さを420mmと誤認しやすいので注意
- PDF.jsの`cMapUrl`指定は必須（日本語CID化フォントの正しい抽出に必要）。パスは相対パス（`vendor/pdfjs/cmaps/`）にすること。絶対パス（`/vendor/...`）だとGitHub Pagesのようなサブパス配信で壊れる
- 一部のPDF帳票（出力元システムの仕様）は全角文字が1文字ずつ空白を挟んで描画される。メタ情報・項目名の抽出時は空白除去（`replace(/\s/g, '')`）を通してから判定・出力すること
- 列境界（隣接ラベルのx座標の中点）を使った値の列振り分けは、値の文字列が列幅より長い場合に隣の列へ値がはみ出すことがある（工期→工事担当への混入など）。汎用的なギャップ検出による解決は誤動作リスクが高いため、業務知識を使ったピンポイントな後処理で対応している
- `window.close()`はスクリプトで開いたタブ以外では多くのブラウザが無視する（ブックマーク・URL直接アクセスで開いた場合は高確率で閉じられない）。閉じられなかった場合の案内文言を必ず表示すること
- キャッシュ対策として`index.html`の`<link>`/`<script>`に`?v=YYYYMMDD`のクエリ文字列を付与している。CSS/JSを変更したら必ずこの値も更新すること

## 既知の制限

- **本人確認の強制力はない**: ログイン機構・サーバー側のユーザー管理がなく、ブラウザ内で完結する構成のため、「本人でなければ物理的に押印できない」というアクセス制御は実現できない。検印フローのロックは「通常操作では順番を飛ばせない」というソフトロックに留まる
- 検印フローは現状「担当者→工事長」までのβ版運用（それ以上は紙ベースでの提出を想定）
- 得意先名など、値が列幅より長い場合の列振り分けミス（列境界をまたいだ値の誤割当）が一部の帳票フォーマットで残っている
- PWAのファイルハンドラ（ダブルクリックで自動起動）はChrome/Edge等の対応ブラウザ・インストール済み環境でのみ動作。Safari/Firefox等では手動の「ファイルを開く」ボタンでのフォールバックとなる

## 他のPC・他のユーザーで開発を続ける場合

```bash
git clone https://github.com/shar19-ops/genka-webapp.git
cd genka-webapp
npm install
npm start   # http://localhost:5173
```

GitHubへの変更を書き戻す（push）には、このリポジトリのコラボレーターとして追加してもらうか、フォーク＋プルリクエストの運用にしてください。
