## 概要

株式会社雄電社の紙/Excelベース「原価計算書」をデジタル化するWebアプリ。
既存の帳票PDF（縦書き・特殊レイアウト）を取り込んでブラウザ上で編集し、
今後の支出予定や追加工事予算を任意入力すると自動で小計・合計・粗利率を再計算、
A3用紙相当の印刷/PDF出力を行う。PC主体、iPadでも利用可能。

## 2. PDF取込ロジック（pdf-parser.js）

帳票PDFは**テキストの格納順序が視覚上の並びと一致しない**ため、座標(x,y)ベースで行を再構成してから列を読み取る方式。

### ポイント
- **CMap必須**: 日本語CID化フォントを正しく抽出するため `cMapUrl: '/vendor/pdfjs/cmaps/'` が必要
- **ページ回転対応**: `page.rotate` がかかっている帳票があるため、`viewport.transform` 適用後の座標を基準にする（生のtransform値をそのまま使うと座標がズレる）
- **行クラスタリング**: y座標の差が `Y_TOL = 2.2` 以内のテキストを同一行とみなす
- **列境界の推定**: ヘッダーラベル行のx座標間の中点を列境界とすることで、値の左詰め開始位置ズレによる誤割当を防止
- **可変長数値列に対応**（帳票バージョン差異）:
  - 4列: 実行合計/原価合計/支払額/未払額（予算残額は当方で算出）
  - 5列: 上記＋予算残額
  - 6列: 上記＋今後の支出予定（システム側で既に保持している版）
- **セクション分割**: `(小計)` 行の出現位置で items を分割 → 3番目のグループをさらに `O00`（現場経費）とそれ以外（工事費/労務）に再分割

## 3. 主要な計算ロジック（app.js）

- `sabikigo`（差引後残額）= (予算残額 + 追加工事予算) − 今後の支出予定
- `formatRate()`: 粗利率は**小数点第2位以下を切り捨て**て小数点第1位まで表示（`Math.floor(rate*10)/10` → `.toFixed(1)+'%'`）
- サマリーカード: 契約金額/実行金額/実行粗利率、予定追加工事契約金額/追加工事実行金額/追加工事粗利率、最終原価/最終粗利率の3ペア＋最終系2枚
- 金額入力欄（契約金額・予定追加工事契約金額など）はカンマ区切り表示（`type="text"` + `yen()`整形をblur時に適用、focus時は`input.select()`で全選択）

## 4. 印刷・PDF出力まわりでハマった点

**最重要**: A3横（landscape）は **幅420mm×高さ297mm**。高さを420mmと誤認して詰まった箇所が複数あった。

| 問題 | 原因 | 解決 |
|---|---|---|
| ヘッダーが消える/重なる | `position: fixed` はChromiumのヘッドレス印刷パイプラインで不安定 | ネイティブ `<thead>` に統合（タイトル+メタ+列見出し）。ブラウザ標準の「各印刷ページで自動repeat」機能を利用 |
| 小計が毎ページ出る | `<tfoot>` はページ毎に繰り返される仕様 | セクションごとの `<tbody>` 内に `.subtotal-row` として持たせる方式に変更 |
| 印刷ページが4→本来2ページのはずが増える | `<input>` の `min-height:40px` が印刷時も適用され行高が肥大化 | `@media print` で `min-height:0; height:auto` を上書き |
| ページ先頭で判子欄がズレる | ページ送り直後の要素は `margin-top` が潰れる（margin collapsing） | `padding-top` または `table-cell`+`vertical-align:bottom` に置き換え |
| サマリーカードの右寄せが効かない | `.summary-pair-grid` が `flex:1 1 480px` で余白を吸収してしまう | `flex:0 0 auto` に変更し、`.summary-final-group` の `margin-left:auto` で真の右寄せを実現 |
| **本番データで6枚1行+final-groupが縦積みになる不具合** | `@media (max-width:900px)` のレスポンシブ規則が、印刷時もブラウザウィンドウ幅を基準に評価され誤発火 | `@media print` 内で `!important` によりgrid-template-columnsを強制指定 |
| ヒント文言「(任意入力)」がラベルと別行に | `<label>text <span>hint</span><input></label>` が `flex-direction:column` の子として、生テキストノードが独立した匿名flexアイテムになっていた | ラベル文言とヒントを `<span class="label-text">` で一つにまとめてラップ |

### 印刷用CSSの要点
- `@page { size: A3 landscape; margin: 12mm; }`
- 大項目・小計行のボーダーは太く維持、通常の項目行は `border-width: 0.4pt` の細罫線に
- 黒背景の強調列（col-tsuika/col-yotei）は `border-left/right` に `!important` を付けて細罫線ルールに負けないようにする
- `border-collapse: collapse` 前提でボーダー太さのカスケードを制御

## 4.5 検印欄（デート印）機能（2026-07-11追加）

`見積書入力アプリ(雄電社様式)`の社員用アプリに実装済みの担当者確認印（見積書入力アプリ(雄電社様式)参照）を移植。印刷出力の右下にあった**ラベルなしの空白判子欄6つ**（`.stamp-cell`、各2cm角、従来は物理ハンコ用）を、6個すべてデジタルのデート印に対応させた。

### 実装方式
- `common-libs/date-stamp.js`（丸型3段デート印描画ライブラリ、`DateStamp.draw(ctx, cx, cy, options)`）をそのまま`public/date-stamp.js`としてコピー。genka-webappはCSS絶対配置オーバーレイ方式ではなくDOM+`window.print()`方式のため、オフスクリーンcanvasで描画→`toDataURL()`でPNG化→`.stamp-cell`に`<img>`として挿入、というシンプルな差し込みで対応
- 6欄それぞれに「苗字（10文字まで）／役職（プルダウン9択）／確認日」の入力欄を`.project-header no-print`内に新設（`#stampInputGrid`）。苗字が空欄の欄は画像を生成せず、物理ハンコ用の空白のまま残る
- 役職9択: `（なし）/副長/課長/工事長/支店長/部長/本部長/取締役/常務/本店長`。色は副長・課長・(なし)＝黒、工事長以上＝朱（支払伝票PJ・見積書アプリと同一ルール）
- 役職なし→苗字を1文字ずつ上下に自動分割。役職あり→上段に苗字・下段に役職
- **見積書アプリ版との差異**: 見積書アプリには「支店長」選択時に事業所名を使う特例があるが、genka-webappには事業所（支店）を表すデータ項目が存在しないため、この特例は移植せず全役職共通ルールで統一した
- 確認日を空欄のまま苗字だけ入力した場合は、当日の日付を自動的に使用する（`todayIso()`フォールバック）
- `currentProject.stampSlots`（6要素の配列、`{name, title, date}`）としてlocalStorageに保存。PDF再取込時も既存の入力値を引き継ぐ（`tsuikaKoujiKeiyaku`と同じ扱い）

### 検印フロー（押印の順番ロック）機能追加（2026-07-11・同日中）
ユーザーからの要望：「押印欄を勝手に押印できないようにしたい。担当者欄は誰でも押印できるが、確認欄に押印するには一度保存して確認者が保存されたファイルを開かない限り押印できないようにしたい。ただし確認者が必ず4人いるわけではないので、確認者が押印し次のフローへ進む際に承認者を選べるようにしたい」

**前提として伝えたこと**: genka-webappにはログイン機構・サーバー側のユーザー管理がなく、ブラウザのlocalStorageのみで動く構成のため、「本人でなければ物理的に押印できない」という強制力のあるアクセス制御は実現不可能（devtoolsで誰でもデータを書き換えられる）。実現できるのは「通常のUI操作では順番を飛ばせない」というワークフロー的なソフトロックのみである旨を説明し、ユーザーの了承を得た上で実装。

### 実装内容
- `currentProject.stampFlow = { stage, skipped }` を新設。`stage`は現在アクティブな欄のインデックス（0=担当者、1〜4=確認、5=承認、6=すべて完了）、`skipped`は使わなかった確認欄のインデックス配列
- 欄の状態は`stampSlotState(i)`で判定: `active`(現在入力可能)/`done`(押印済み・disabled)/`locked`(未解放・disabled)/`skipped`(未使用・disabled)。`active`以外はすべて`<input disabled>`にして編集不可にする
- 各状態をバッジ表示（🔒未解放／押印済み／未使用）、行全体をCSSで薄く(`opacity`)して視覚的にも分かりやすくした
- フロー操作ボタン（`#stampFlowActions`、`renderStampFlowActions()`で内容切り替え）:
  - stage=0（担当者）: 「担当者欄を保存して確認へ進む」1つのみ
  - stage=1〜4（確認、4段目は「もう1人確認を挟む」を無効化）: 「もう1人確認を挟む」と「承認へ進む」の2択。後者を選ぶと`stage+1`〜4のインデックスを`skipped`に追加し、承認欄(5)を即座に解放する→これが「確認者が可変」の実現方法
  - stage=5（承認）: 「承認欄を保存して完了」
  - stage=6: 「✓ すべての検印が完了しました」のメッセージのみ
- 各ボタンは`advanceStampFlow(action)`で処理し、`saveProject()`を**即時同期呼び出し**（800msデバウンスの`scheduleAutosave`とは別）することで「進む＝保存」を保証している
- 名前入力中はボタンのdisabled状態だけを`refreshStampAdvanceButtons()`で軽量更新し、グリッド全体を再描画しない（キー入力中にフォーカスが飛ぶのを防止するため）
- 出力（印刷）側は無改修。`stampFlow`の状態に関わらず、`stampSlots[i].name`が空なら従来通り画像を生成しないだけなので、`locked`/`skipped`状態の欄は自然に空欄の判子欄として印刷される
- 旧データ（このフロー機能導入前に自由入力されていたプロジェクト）との互換: `ensureStampFlow()`で`stampFlow`が無ければ、先頭から連続して入力済みの欄までを`done`とみなす簡易移行処理を実装

### 検証
Playwrightで一連のフロー（担当者入力→進む→確認1入力→もう1人確認を挟む→確認2入力→承認へ進む(確認3,4が自動スキップされることを確認)→承認欄入力→完了）を実行し、各段階で意図通りの欄がロック/解放されること、ロック中の`<input>`が実際に`disabled`属性を持ち値を保持したまま編集不可であること、印刷PDFにも最終的な押印内容（本部長太郎=承認/佐藤=確認2/山田=担当者、確認3・4は空欄）が正しく反映されることを確認済み。

### 差し戻し機能の追加（2026-07-11・同日中）
ユーザー要望「検印欄に差戻しを追加し、前のフローへ戻れるようにする」に対応。事前に2点確認: (1) 差し戻し先は「1つ前の実段階のみ」（任意の段を選べる方式ではない）、(2) 差し戻し元・差し戻し先の押印データは両方クリアして再入力を求める（値を残して編集可にするだけ、ではない）。いずれも推奨案（デフォルト）で確定。

#### 実装内容
- `prevRealStage(stage)`: 指定した段より前で直近の「スキップされていない実段階」のインデックスを返すヘルパー。`flow.skipped`に含まれる段（承認へ早く進んだ際に未使用となった確認欄）は飛ばして遡る
- `advanceStampFlow('reject')`: `prevRealStage(現在の段)`が見つかれば、①現在の段の入力欄をクリア、②差し戻し先の欄もクリア（再入力必須にする）、③`flow.skipped`から差し戻し先より後ろのインデックスを除去（スキップ判断をやり直せるようにする）、④`flow.stage`を差し戻し先に更新。担当者欄（段0）より前はないため、`prevRealStage(0)`が-1を返す場合はボタン自体を出さない
- 「完了」状態（stage=6）からも差し戻し可能にした。この場合の1つ前の実段階は常に承認欄（段5）になる（承認欄は仕組み上スキップされないため）。つまり「完了→差し戻す」を押すと承認欄がクリアされ再度アクティブになり、そこからさらに「差し戻す」を押すと確認欄側へ連鎖的に遡れる
- ボタン文言は「差し戻す（欄○へ）」とし、○には遷移先の1始まりのインデックスを表示（「確認」ラベルが複数あり紛らわしいため、役職名ではなく欄番号で明示）
- `.btn.danger`（枠線・文字色を`var(--danger)`の朱色に）を新設しボタンに適用
- `refreshStampAdvanceButtons()`（入力中の軽量再描画）では差し戻しボタンだけは苗字未入力でも常に押せるよう除外（自分の欄を埋めていなくても差し戻しは可能にすべきため）

#### 検証

#### 副次的に発見した問題（今回は未対応・要フォローアップ）

#### 現状の自動反映フィールド（調査結果、ユーザーへの回答用メモ）

### 実PDFでの「契約金額と入金額しか出ない」不具合の原因特定・修正（2026-07-11・同日中）
ユーザーが実際に社内サーバー（`http://s010web3/genka/pdf/...`）からURL取込した本番PDFのスクリーンショットで、「契約金額・入金額以外のメタ情報（工番・得意先名・件名等）が全部空欄になる」という不具合が判明。上記の「副次的に発見した問題」で見つけた別サンプルPDFの不具合と**同じ根本原因**であることが、このPDFを実際にダウンロードして`GenkaPdfParser.extractLines()`で直接調査した結果判明した。

**根本原因**: この帳票フォーマットは、見出し行・値の全角文字が**1文字ごとに半角スペースを挟んで**PDF内に描画されている（例: `"(工 番 ) (得 意 先 名 )"` 、値側も `"鹿 島 建 設 ㈱..."`）。これは行の座標クラスタリングが生んだ副作用ではなく、PDFの文字コンテンツ自体に空白文字が挟まっている（＝出力元システムの仕様）。

- `parsePdf()`内のメタ見出し行検出 `allRows.findIndex((r) => r.text.includes('工番') && r.text.includes('得意先名'))` が、空白を考慮していなかったため一致せず、`meta`が終始`null`のまま → 契約金額・入金額（footerの正規表現は`compact = line.replace(/\s/g,'')`で空白除去済みだったため無事だった）以外すべて空欄になっていた

**修正内容**（`public/pdf-parser.js`）:
1. 見出し行検出を空白除去後の文字列で判定するよう変更（`compact.includes('工番') && compact.includes('得意先名')`）。折り返し得意先名の継続行チェック（`項目名`を含むかの判定）も同様に修正
2. `buildMetaFromColumns()`: 抽出した値からも空白を除去（`col.value.replace(/\s/g, '').trim()`）。日本語の工番・会社名・件名等に意味のある空白は通常ないと判断し除去
3. `parseItemRow()`: 項目名からも同様に空白除去（`各 戸 盤`→`各戸盤`のような文字化けを解消）

**検証**: このPDFを実際にダウンロード(`curl`)し、Playwrightで`GenkaPdfParser.parsePdf()`を直接呼び出して確認。修正前は`meta`が空、修正後は工番="25-0183"、得意先名・件名・完成計上月・項目名（例:「各戸盤」）が正しく抽出されることを確認。契約金額(8,030,000,000)・入金額(0)は修正前後とも正常。

**残課題（未修正・要フォローアップ）**: 得意先名が「鹿島建設㈱東京建築支」で途切れる（本来は「支店」まで続くと思われる）、工期の値の末尾が工事担当欄に混入する、といった**列境界をまたいだ値の誤割り当て**が残っている。これは「値の文字列が想定より長く、列境界（隣接ラベルのx座標の中点）を越えてしまう」という別種の問題で、単純な空白除去では直らない。取込プレビュー画面で該当欄は手動修正可能なため運用は継続できるが、抜本的な解決には列境界の判定ロジック自体の見直しが必要。次回対応時の出発点として記録。

### 工期→工事担当への値混入バグの根本原因特定・修正（2026-07-11・同日中）
ユーザーが実際の取込プレビュー画面のスクリーンショットで再現を確認（工事担当欄に「/06/26根間和佳」と表示）。上記「残課題」に書いた問題を実際にPDFの文字座標(x, str, width)レベルまで掘り下げて調査し、ピンポイントで修正した。

**精密調査で判明した内容**: `pdfjsLib`のテキスト抽出では、この帳票の「純粋な空白1文字」のtext item（列と列の間の余白を表現するもの）が、実際の見た目の空白幅に対応する**異常に大きい`width`値**（数百〜千数百pt）を持って返ってくる。一方、実データの文字item（漢字の連続文字列や数字）の`width`は正確。座標を精密に計測した結果、工期ラベルと工事担当ラベルのx座標から計算される列境界の中間点(≈692.08pt)が、値「25/09/01～30/06/26」の描画範囲(581.52〜726.98pt)の**ちょうど途中**（"30"と"/06/26"の間）に来てしまうことが直接の原因と特定。値の文字列自体が列幅の想定より長いため、`extractMetaByColumns()`の中間点ベースの列振り分けでは原理的に防げないタイプの不具合。

**検討した代替案と却下理由**: 値の文字間ギャップの大小で自然な区切りを検出する案（ギャップベースの再クラスタリング）も検討したが、実測したところ「工期内の日付セグメント間のギャップ(13.65〜19.71pt)」と「工期→工事担当の本当の境界のギャップ(20.85pt)」の差がわずか1pt程度しかなく、閾値ベースの汎用的な検出は別のPDFで簡単に誤動作しうると判断し採用しなかった。

**採用した修正**（`buildMetaFromColumns()`、`public/pdf-parser.js`）: 列境界アルゴリズム自体は変更せず、**症状に対するピンポイントな後処理**を追加。「工事担当（氏名）の値が数字・スラッシュ・～で始まることは実際にはあり得ない」という業務知識を利用し、`meta.tantou`の先頭が`/^([\d/～]+)/`にマッチする場合、その断片を`meta.koki`側に付け戻す処理を追加。汎用的な列境界ロジックに手を入れるより誤検知リスクが低いと判断。

**検証**: 実際のPDFで再テストし、修正前は`koki:"25/09/01～30", tantou:"/06/26根間和佳"`だったのが、修正後は`koki:"25/09/01～30/06/26", tantou:"根間和佳"`と正しく分離されることを確認。実際の取込プレビュー画面のスクリーンショットでも工期・工事担当とも正しい値が表示されることを確認済み。

なお、得意先名が「鹿島建設㈱東京建築支」で途切れる件は、この1件のサンプルだけでは「PDF側のデータ自体がこの値で完結している（＝バグではない）」可能性と「別の列境界問題で末尾が失われている」可能性を区別できなかったため、今回は手を付けていない。次にこの現象が再現した際は、正しい期待値（本来の得意先名がどこまでか）をユーザーに確認してから対応する。

## 9. 社内公開（GitHub Pages化）（2026-07-11）

### 検討の経緯
「社内公開へ進めたい」との依頼を受け、以下の順で要件を詰めた:
1. genka-webappは元々`localStorage`のみでデータを保持しており、検印フロー（担当者→確認→承認）は**複数人・複数PCでの利用が前提**（ユーザー確認済み）なのに、localStorageは1ブラウザに閉じているため機能しないことを指摘
2. 公開先としてSharePointが候補に上がったが、SharePointはNode.js/Expressのようなサーバーサイド実行環境を提供できず、かつ「自由にサーバーを立てる権限がなくSharePointが唯一自分で公開できる場所」という制約（社内サーバー・IT部門の関与なし）があったため、SharePoint単体での実現は非現実的と判断
3. 最終的にユーザーから「GitHubに本体を置き、データ保存先は個人が任意の場所に保存（ファイル）、Teams/メール添付で次の担当者に渡し、受け取った側はファイルを開くと自動でアプリが表示される」という設計が提示された。これは同一プロジェクト内の見積書入力アプリ(雄電社様式)の「業者がJSON保存→社員が開く」という既存パターンと同じ考え方で、サーバー・SharePoint連携ともに不要になる現実的な解

### 実装した機能

#### 1. ファイル保存/読み込み（`public/app.js`）
- `exportProjectToFile()`: `currentProject`全体（検印フロー状態含む）をJSON化し、`原価計算書_${工番}.json`として保存。`showSaveFilePicker`（File System Access API）→ 非対応時は`<a download>`Blobフォールバック、という見積書アプリと同じ二段構え
- `openProjectFromFile(file)`: JSONを読み込み、`koban`と`sections`配列の存在だけ簡易バリデーションしてから`currentProject`にセット・localStorageにも保存（自分のPCの工番一覧にも載る）・`showMainView()`
- UI: トップバーに常設の「ファイルを開く」ボタン＋隠しinput、プロジェクトツールバーに「ファイルに保存（Teams/メール送付用）」ボタンを追加

#### 2. PWA化（ファイルのダブルクリックで自動起動）
- `public/manifest.json`を新設。`file_handlers`で`.json`ファイルとの関連付けを宣言（Chrome/EdgeでPWAインストール後、添付ファイルのダブルクリックで直接起動できるようにするため）
- `public/sw.js`: インストール可能要件を満たすための最小限のService Worker（キャッシュ戦略は持たず全リクエストをそのままネットワークに通す＝更新が常に反映される）
- `public/icons/icon-192.png`・`icon-512.png`: Playwrightでcanvas描画したプレースホルダーアイコン（後で正式な会社ロゴに差し替え可能）
- `app.js`に`window.launchQueue.setConsumer(...)`を追加し、PWA起動時に渡されたファイルを`openProjectFromFile()`へ渡す実装。`'launchQueue' in window && window.LaunchParams`で機能検知し、非対応ブラウザ（Safari/Firefox等）では単に無視される
- 「PWA未インストール時は手動ボタンで確実に開ける」というハイブリッド方針を採用（ユーザー確認済み）。ボタンでの手動オープンは全ブラウザで動作するフォールバックとして常設

#### 3. GitHub Pages（静的ホスティング）対応
サーバー(server.js)なしでも動くよう、以下を修正:
- `node_modules/pdfjs-dist`から動的配信していたpdf.js本体・cmapsを`public/vendor/pdfjs/`配下に**物理コピー**（`build/pdf.js`・`build/pdf.worker.js`・`cmaps/*`）。server.jsの該当する動的static mountは削除（今は`express.static('public')`だけで完結）
- `index.html`・`app.js`・`pdf-parser.js`内の`/vendor/pdfjs/...`という**絶対パス**を`vendor/pdfjs/...`という**相対パス**に修正。GitHub Pagesは`https://shar19-ops.github.io/genka-webapp/`のようにリポジトリ名がサブパスに入るため、先頭`/`の絶対パスだと壊れる（ローカルのserver.jsではルート直下配信だったため気づかなかった）
- `/api/fetch-pdf`（URL取込プロキシ）はサーバー機能のためGitHub Pagesでは動作しない。起動時に`fetch('api/fetch-pdf')`を試し、**404なら「URLから読み込む」タブを自動的に非表示**にする分岐を追加（ローカルであれば同エンドポイントは400を返すため区別できる）。ローカルで`npm start`する場合は従来通りURL取込も使える

### GitHub公開の経緯・設定
- リポジトリ名は当初「原価計算書App」で依頼されたが、日本語リポジトリ名だとPages URLがパーセントエンコードされ読みにくくなること、他の社内アプリ（`shiharai-denpyo`等）が全て英数字名で統一されていることを説明し、`genka-webapp`に変更してもらった

### 12.2 このセッションで繰り返し踏んだCSSバグパターン（教訓）

**`[hidden]`属性より明示的な`display`指定が優先されてしまう問題**を、`.warning-btn`・`.carried-over-notice`・`.reimport-carry-over-row`の3箇所で立て続けに踏んだ。`.foo { display: flex; }`とだけ書いて`hidden`属性で表示制御しようとすると、HTML5の`[hidden]`のUAスタイル（`display:none`）と同じ詳細度のため、後勝ちで開発者側の`display:flex`が勝ってしまい**常に表示された状態**になる。今後この手のトグル要素を作る際は、`display`を明示指定するなら必ず`.foo[hidden] { display: none; }`をセットで書くこと。

**`<th colspan>`に`display:flex`を直接指定するとcolspanの列幅計算が崩れる**（Chromiumの`table-layout:fixed`実装の挙動）。ヘッダー行内でメタ情報とラベルを左右に振り分けたくて`<th colspan="9"><span>...</span><span>引用元</span></th>`に`display:flex`を付けたところ、そのセルの実際の描画幅がcolspan全体ではなく先頭列1列分にまで縮んでしまう不具合が発生（一般材警告書・原価計算書本体の両方で発生・修正）。対策は`<th>`自体はtable-cellのまま維持し、中に`<div class="xxx-inner">`を1枚挟んでそちらだけflex化すること。

**`<col style="width:0">`は`table-layout:fixed`で完全には列を消せない**。「原価計算書の今後の支出予定額」参考列を印刷時に非表示にする際、`col`要素の幅を0にする・`!important`で上書きする・IDセレクタで特定するなど複数のCSSアプローチを試したが、ブラウザが最小幅を確保してしまい常に一定の幅で描画され続けた（原因は特定できず）。最終的に**CSSでの非表示を諦め、`beforeprint`/`afterprint`イベントでJS側が列（`<col>`/`<th>`/`<td>`）自体をDOMから出し入れする方式**に切り替えて解決。「見た目だけ消す」より「そもそもDOMに存在させない」方が確実、という教訓。

**colgroupでmm指定と%指定を混在させると、ブラウザが全列を比例スケーリングしてしまう**。区分名列を45mm・今後支出予定額列を40mmに固定したいという要望に対し、他の列はそれまで通り%のまま残したところ、指定した45mm/40mmが正確に反映されず値が変動した。全列をmm単位に統一し、かつ表自体を`width:auto`にして「指定した列幅の合計＝表の実際の幅」になるようにしたところ解決した。

**`@page { margin }`はオンスクリーンの`emulateMedia('print')`描画には反映されない**（実際のPDF生成時のみ有効）。Playwrightで座標をmm単位で正確に検証する際、当初ビューポートをA3全体（420mm×297mm）に設定していたため、実際の印刷可能領域（用紙サイズ−左右上下12mmずつ＝396mm×273mm）より広い前提で計算しており、報告した座標がわずかにズレていた。この点に気づいてから、ビューポートを印刷可能領域サイズに合わせて測定し直し、以降はこの方式に統一した。

#### 発生した不具合と根本原因（このセッション最大のハマりどころ）
ユーザーが実際にフォルダを選択しても「画面に何も表示されない」という報告を受け、当初は**OneDriveのFiles On-Demand（オンラインのみファイル）が原因**という仮説を立てて対処（エラーメッセージの改善等）したが、ユーザーがOneDrive外（Dドライブ、デスクトップ）のフォルダでも同じ症状を報告したことで仮説を撤回。

**真の原因**: `change`イベントハンドラ内で`const files = e.target.files; e.target.value = '';`という順序で書いていたが、**`input.files`（FileList）は`input.value`と連動したライブ参照**であり、`value`をリセットした瞬間に、既に変数へ代入したはずの`files`の中身も一緒に空になっていた。フォルダの場所やブラウザの挙動とは無関係な、単純な自分のコードの実装ミスだった。

**修正**: `const files = Array.from(e.target.files || []);`で即座に配列へスナップショットしてから`value`をリセットするよう順序を修正。

**検証方法**: Playwrightの`webkitdirectory`フォルダ選択automationは（許可パス制限とは別に）ディレクトリ選択そのものが正しく動作しない制限があり、実際のフォルダピッカー操作は自動テストできなかった。代わりに、同一オリジンからPDFを`fetch`して`File`オブジェクトを構築し、`DataTransfer`経由で`input.files`にセット→`change`イベントを`dispatchEvent`する方式で、実際のイベントフロー（`value`リセットのタイミングを含む）を再現してバグを検出・修正確認した。

**教訓**: `<input type="file">`の`change`ハンドラで`files`を扱う際は、必ず`Array.from()`等で独立した配列に変換してから`input.value`をリセットすること。既存の`#fileInput`/`#openFileInput`のハンドラは`files[0]`で単一の`File`オブジェクトを直接取り出していたため（`File`自体は独立したBlobで影響を受けない）同じ問題を踏んでいなかった、という違いも判明した。

### 13.2 manual.htmlへの反映方法（大容量ファイルの扱い）
`manual.html`は埋め込みbase64画像により約1MB・1行が数十万トークンに達する行が複数あり、Readツール・Editツールで直接開くと単純な範囲読み込みでもトークン上限に抵触する。対策として：
- 構造把握は`Grep`（`-n`, `-o`付き）でセクション見出し・`<figure>`タグ・`alt`属性など**必要な断片だけ**を抽出し、巨大なbase64行そのものは一切読み込まないようにした
- 実際の書き換えはEdit/Writeツールではなく、**Node.jsスクリプトをBashツール経由で実行し、`fs.readFileSync`→文字列置換（アンカー文字列の完全一致）→`fs.writeFileSync`で行った**。base64データはNode側で画像ファイルを都度読み込んでエンコードし、会話コンテキストには一切base64文字列を出力しない設計にした
- 注意点: `manual.html`は改行がCRLF(`\r\n`)。アンカー文字列をJSのテンプレートリテラル（`\n`区切り）で書いていたため最初は不一致で失敗し、`crlf()`ヘルパーで`\n`→`\r\n`変換してから置換する形に修正した
